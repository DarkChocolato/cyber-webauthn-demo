อธิบายการทำงานอย่างละเอียด (System Flow)

ไดอะแกรมและคำอธิบายขั้นตอนการลงทะเบียนและการเข้าสู่ระบบ FIDO2 / WebAuthn

ระบบยืนยันตัวตนไร้รหัสผ่าน FIDO2/WebAuthn จะทำงานในลักษณะของ Asymmetric Cryptography (การเข้ารหัสแบบสมมาตรแบบกุญแจคู่) โดยอุปกรณ์ของผู้ใช้จะเก็บกุญแจส่วนตัว (Private Key) ไว้ในชิปความปลอดภัยอย่างมิดชิด และระบบของเรา (ฐานข้อมูลจำลอง) จะเก็บเฉพาะกุญแจสาธารณะ (Public Key)

🔑 1. ขั้นตอนลงทะเบียน (Registration)

  1. ผู้ใช้ป้อนชื่อผู้ใช้และข้อมูลส่วนตัวในแบบฟอร์มสมัครสมาชิก
  2. ระบบตรวจสอบว่าชื่อผู้ใช้ไม่ซ้ำ แล้วสร้างรหัสท้าทาย (Challenge) แบบสุ่มเพื่อป้องกันการโจมตีซ้ำ (Replay Attack)
  3. เบราว์เซอร์เรียกใช้ navigator.credentials.create() เพื่อสั่งให้ชิปฮาร์ดแวร์ในอุปกรณ์สแกนนิ้วสร้างคู่กุญแจ
  4. ผู้ใช้ผ่านด่านสแกนลายนิ้วมือ/ใบหน้า หรือ PIN เพื่อยืนยันว่าเป็นเจ้าของเครื่องจริง
  5. ชิปความปลอดภัยภายในเครื่องจะส่ง Credential ID และกุญแจสาธารณะ (Public Key) กลับคืนมา
  6. เบราว์เซอร์บันทึกข้อมูลและเซฟลงใน LocalStorage เพื่อใช้เป็นฐานข้อมูลหลักจำลอง

🔓 2. ขั้นตอนเข้าสู่ระบบ (Authentication)

  1. ผู้ใช้ป้อนชื่อผู้ใช้งาน (Username) ที่เคยสมัครไว้
  2. ระบบดึง Credential ID ที่เคยบันทึกไว้คู่กับบัญชีผู้ใช้ขึ้นมาจาก LocalStorage
  3. ระบบสุ่มข้อมูล Challenge รอบใหม่ เพื่อความปลอดภัยและเป็นเอกลักษณ์ในเซสชันล็อกอินนั้น ๆ
  4. เบราว์เซอร์เรียกใช้ navigator.credentials.get() พร้อมระบุ Credential ID เพื่อระบุอุปกรณ์เป้าหมาย
  5. ผู้ใช้สแกนลายนิ้วมือ/ใบหน้า เพื่ออนุญาตให้ชิปความปลอดภัยลงลายมือชื่อดิจิทัล (Digital Signature) ลงบน Challenge
  6. ชิปส่งผลการลงลายมือชื่อกลับคืนมาให้เบราว์เซอร์ เพื่อยืนยันว่าสแกนผ่านและกุญแจถูกต้อง
  7. เบราว์เซอร์ยืนยันความสอดคล้อง ดึงข้อมูลชื่อ นามสกุล และเบอร์โทรศัพท์ของผู้ใช้ขึ้นแสดงในหน้าสำเร็จทันที

การทำงานในโปรเจกต์นี้

  • การเข้ารหัสข้อมูล: ข้อมูลไบนารี (ArrayBuffer) จากชิปความปลอดภัยถูกแปลงเป็น Base64 เพื่อจัดเก็บลงใน LocalStorage ของเบราว์เซอร์
  • การลงทะเบียน: อุปกรณ์เครื่องจะสร้างคู่กุญแจ (Public/Private Key pair) โดยกุญแจสาธารณะจะเก็บไว้บนเครื่องจำลอง (LocalStorage) และกุญแจส่วนตัวอยู่ในชิปฮาร์ดแวร์ของผู้ใช้
  • การยืนยันตัวตน: เมื่อผู้ใช้เข้าสู่ระบบ อุปกรณ์ของผู้ใช้จะลงลายมือชื่อดิจิทัล (Digital Signature) บนข้อมูล Challenge แล้วส่งกลับมาตรวจสอบความสอดคล้อง

การนำไปประยุกต์ใช้ในระบบจริง

  • Backend จริง: ฝั่งเซิร์ฟเวอร์จะต้องใช้ไลบรารีช่วยตรวจสอบ เช่น fido2-lib (Node.js) หรือ WebAuthn SDK ของภาษาอื่นๆ
  • การทวนสอบ (Verification): Backend จะสร้าง Challenge ที่ปลอดภัยแบบสุ่มจริง และทวนสอบลายเซ็นที่ส่งกลับมาจากเบราว์เซอร์ด้วยกุญแจสาธารณะที่บันทึกไว้ในฐานข้อมูล
  • ความปลอดภัย: ป้องกันการโจมตีแบบ Replay Attack และการทำ Phishing ได้ 100% เพราะกุญแจจะผูกกับชื่อโดเมน (Relying Party ID) เสมอ
